# Jasper 平台化（托管服务）可行性与方案

> 目标：评估把 Jasper 从「个人自托管的单实例应用」做成「多用户的托管服务（SaaS）」的可行性、成本量级，以及代码需要往哪个方向调整。
> 本文只做方案论证与路线规划，不含实现代码。结论先行，细节在后。

## 0. TL;DR（结论先行）

- **可行，但要区分"平台形态"**：不要往纯 FaaS（AWS Lambda / Cloudflare Workers）方向靠——Jasper 是常驻进程 + 内存索引 + 本地 SQLite + SSE 长连接，天然不是无状态函数模型。应选**「按量付费的容器」**这一档（Fly.io Machines / Google Cloud Run / Cloudflare Containers 等）。
- **架构方向：每租户一实例（instance-per-tenant），不是单进程多租户**。Jasper 现在是彻底的单租户设计（见 §2），改成"一个进程服务所有用户"要重写核心（`AppState` / `ConfigStore` / 内存索引 / 插件宿主全部要按租户切分），代价极大且违背"后端常驻 ~10MB"的初衷。相反，"一租户一份 Jasper 实例 + 一层控制面"几乎不用改 Jasper 本体。
- **推荐落地形态**：**Fly.io Machines**（`one app per customer` 官方模式）+ **S3 对象存储做每租户的数据真源**（复用已有的 `s3-storage` 插件）+ **Fly 持久卷仅作本地缓存**。这样每个实例都是可抛弃的（挂了/被回收就重建，真源在 S3），天然支持 scale-to-zero 和备份。
- **成本量级**：基础设施单租户约 **$0.3–1/月（闲置为主，scale-to-zero）** 到 **$2–3.4/月（常开）**；总成本 ≈ 单租户 × 活跃租户数。**真正的成本大头不是服务器，是"从零开始"的控制面工程（鉴权/开通/路由/计费）和托管他人私密数据的运营与合规责任**——这些目前 Jasper 一行都没有。
- **对 Jasper 本体的最小改动**：加 `SIGTERM` 优雅退出（配合 auto-stop）、把 `/api/status` 当健康检查、确保只经 Fly 私网暴露（不直接上公网，因为**内部无鉴权**）。真正的新代码在控制面那一侧。

## 1. 平台形态：三档模型该选哪一档

| | 传统 VM / K8s（IaaS） | **按量付费容器（Serverless 容器）** | 纯 FaaS（Lambda/Workers） |
|---|---|---|---|
| 交付物 | 自管整机 | **标准 Docker 镜像** | 符合平台运行时的函数 handler |
| 计费 | 按时长常收，闲时照扣 | **按 vCPU-秒 + 内存 GB-秒，scale-to-zero 闲时近零** | 按调用次数 + 执行毫秒 |
| 常驻状态/长连接 | ✅ | ✅（实例存活期内） | ❌ 无状态、长连接差、单次时限 |
| 与 Jasper 契合度 | 能跑但闲时费钱、要自己做扩缩容 | **最契合** | **不契合**（要重写架构） |
| 代表产品 | EC2 / GCE / 自建 K8s | **Fly.io Machines / Cloud Run / Cloudflare Containers / App Runner / Railway** | AWS Lambda / Cloudflare Workers |

**为什么 Jasper 不能上纯 FaaS**（详见早先分析，摘录）：
- 常驻内存索引：启动时把整库扫进 `Arc<RwLock<Library>>`（`server/src/main.rs:96-124`, `server/src/api.rs:47`），后续请求吃这份内存索引，不是每请求无状态计算。
- 本地 SQLite 状态：`config.db` / `cache.db` 是 rusqlite 文件（`server/src/config.rs`, `server/src/cache.rs`），FaaS 文件系统临时且隔离。
- SSE 长连接：`GET /api/events`（`server/src/events.rs`）是常驻推流，Workers/Lambda 对长连接支持很差。
- 插件系统落盘：`.jplug` 解压到 `<config>/plugins/<id>/`，需要持久文件系统。

**候选容器平台横评**（同为"按量付费容器"档）：

| 平台 | 持久盘 | scale-to-zero | 单实例绑定 | 对 Jasper 的坑 |
|---|---|---|---|---|
| **Fly.io Machines** | ✅ 原生持久 Volume（重启不丢） | ✅ auto_stop/auto_start | ✅ 一 app 一客户官方模式 | 最省事；Volume 24/7 计费 |
| Google Cloud Run | ⚠️ 需接 GCS FUSE / Filestore，较绕 | ✅ | 多实例扩容会分裂状态，要 `max-instances=1` | 端口读 `$PORT`（Jasper 读 `JASPER_PORT`，可对齐） |
| Cloudflare Containers | ❌ **磁盘完全临时**，sleep 后清盘 | ✅ sleepAfter | ✅ 一实例一 Durable Object | 状态必须外置到 D1/R2，改动最大 |

> Cloudflare Containers 官方 FAQ 原话：容器磁盘完全 ephemeral，实例 sleep 后下次唤醒是从镜像重建的"全新盘"。这对 Jasper 的本地 SQLite 持久化是致命的，除非把 `ConfigStore`/`CacheStore` 改成读写 D1。**因此首选 Fly.io。**

## 2. 现状盘点：Jasper 是彻底的单租户设计

要评估改造量，先看清现状。Jasper 的每一处状态都假设"整个进程只服务一个人、一个数据源"：

| 关注点 | 现状（代码位置） | 平台化的问题 |
|---|---|---|
| **鉴权** | **完全没有**。`api::router()` 只挂了 `guard_read_only` + CORS `permissive` + Trace（`server/src/api.rs:64-91`），没有任何身份/会话中间件 | 直接暴露到公网 = 任何人可读写全部笔记 |
| **应用状态** | `AppState` 单份：一 library 索引、一 storage、一 config、一 cache、一 plugin host（`server/src/api.rs:45-58`, 构造在 `main.rs:150-158`） | 无"当前用户是谁"的概念，无法在同进程内区分租户 |
| **配置库** | 单个 SQLite 文件 `config.db`（`server/src/config.rs`），一进程一份，明文存 `webdav_pass`/`api_key` | 多租户需按租户切分配置与密钥，且密钥明文不可接受 |
| **数据源** | 一个进程绑定一个数据源（local/webdav/plugin），`RwLock<Option<Arc<dyn StorageBackend>>>` | 每租户需各自的数据源与隔离 |
| **缓存库** | `cache.db` 按 `source_key` 隔离，但仍是一进程一文件（`server/src/cache.rs`） | 同上 |
| **插件** | 安装到 `<config>/plugins/<id>/`，能力授权、`host:http`/`host:ai` 出网（`server/src/plugins/`） | 托管场景下让租户装任意 wasm 且能出网，是风险面 |
| **参数化** | ✅ 已用环境变量全参数化：`JASPER_CONFIG_DIR`/`JASPER_HOST`/`JASPER_PORT`/`JASPER_SOURCE`（`main.rs` 头注释） | **这是好消息**：起多个隔离实例几乎零改动 |

**关键判断**：前 6 行说明"单进程多租户"要伤筋动骨；最后 1 行说明"多进程、每租户一实例"几乎免改。方向由此确定。

## 3. 架构方案：每租户一实例（instance-per-tenant）

### 3.1 为什么不选"单进程多租户"（方案 B，不推荐）

改造清单（仅供说明代价，**不建议做**）：
- `AppState` 要变成 `Map<TenantId, TenantState>` 或每请求解析租户 → 触及几乎所有 handler。
- `config.db`/`cache.db` 要么加 `tenant_id` 列重构 schema，要么一租户一库。
- **内存索引按租户各一份**：`常驻 ~10MB` 会随活跃租户数线性膨胀，一个进程扛不住多少租户，反噬了轻量初衷。
- 插件宿主按租户隔离（插件安装是 per-config-dir 的）。
- 全套鉴权 + 租户解析中间件。

一句话：等于重写核心假设，且把"轻量单实例"的优点亲手扔掉。

### 3.2 推荐方案 A：每租户一实例 + 控制面 + 对象存储真源

```
                      ┌────────────────────────────────────────────┐
   用户浏览器 ──TLS──▶ │  控制面 Gateway（新建，唯一公网入口）         │
                      │  · 鉴权/会话（谁登录了）                      │
                      │  · 租户→实例 路由                            │
                      │  · 开通/回收（调 Fly Machines API）          │
                      │  · 计量/计费（Stripe）                       │
                      └───────────────┬────────────────────────────┘
                                      │ 认证后经 Fly 私网(flycast) 反代
                       ┌──────────────┼──────────────┐
                       ▼              ▼              ▼
                 ┌───────────┐  ┌───────────┐  ┌───────────┐
                 │ Jasper 实例│  │ Jasper 实例│  │ Jasper 实例│  ← 每租户一个（Fly Machine）
                 │  租户 A    │  │  租户 B    │  │  租户 C    │    内部无鉴权，仅私网可达
                 │ +Volume(缓存)│ +Volume    │  │ +Volume    │    Volume 只存 config.db/cache.db
                 └─────┬─────┘  └─────┬─────┘  └─────┬─────┘
                       ▼              ▼              ▼
                 ┌───────────────────────────────────────────┐
                 │  对象存储 S3（每租户一 bucket/prefix）= 真源  │  ← 复用 s3-storage 插件
                 │  笔记/资源都在这里；实例可随时销毁重建         │    备份/持久性由 S3 负责
                 └───────────────────────────────────────────┘
```

**三个设计支点**：

1. **控制面（新建服务）是唯一公网入口，承担全部身份**。Jasper 实例本身继续"无鉴权"，但**永不直接上公网**，只在 Fly 私网（`.flycast`/`.internal`）内可达；Gateway 认证用户后把请求反代给该用户的实例。这样 Jasper 内部代码几乎不用改（不用现造一套鉴权），身份边界收敛到一处。
2. **S3 对象存储做每租户的数据真源**（关键）。Joplin 数据格式就是一堆文件，`s3-storage` 插件（在 jasper-plugins 仓库）已经能把 S3 bucket 当数据源。于是：**真源在 S3（廉价、持久、易备份），Fly Volume 只是本地缓存，Jasper 实例变成可抛弃的**——被 scale-to-zero 回收、迁移、重建都不丢数据，冷启动时从 `cache.db` 增量重建索引即可。这把"有状态实例"降级成"近乎无状态实例"，是平台化最省心的一招。
3. **一租户一 Fly App/Machine + 一 Volume**，对齐 Fly 官方 `one app per customer` 模式：应用级密钥隔离、独立网络、独立伸缩、独立 auto-stop。

### 3.3 请求生命周期

- **注册**：控制面创建租户记录 → 调 Fly Machines API 创建该租户的 app + machine + volume → 首启用引导环境变量（`JASPER_SOURCE` 指向该租户 S3 前缀 + 凭证）完成初始化。
- **访问**：用户登录控制面 → 控制面按租户路由（子域名 `tenant.jasper.app`，或统一网关按会话反代）→ 命中该租户实例。
- **闲置**：Fly 原生 `auto_stop_machines` 空闲停机；下次请求 Fly 代理 `auto_start` 唤醒（Volume 数据仍在，唤醒后走增量缓存，不用全量拉取）。
- **注销**：控制面调 API 销毁 machine + volume；真源 S3 按保留策略清理或导出。

## 4. 成本模型

### 4.1 单租户基础设施成本拆解（Fly.io，2026 现价）

| 项目 | 单价 | 说明 |
|---|---|---|
| compute `shared-cpu-1x` 256MB | $0.0027–0.0044/小时（≈$1.94–3.14/月常开） | Jasper 常驻 ~10MB，256MB 档绰绰有余；按秒计费，仅"醒着"时收 |
| 持久 Volume | $0.15/GB/月 | **不管实例是否在跑都 24/7 收**；只存 config.db/cache.db，1GB 档足够 |
| 出网流量 | $0.02/GB（NA/EU），入站免费 | 个人用量通常 <1GB/月 |
| 共享 IPv4 | 免费 | 独享 IPv4 才 $2/月，本场景不需要 |
| 固定月费 / 套餐门槛 | **无** | 纯按量；旧免费额度已下线，需绑卡 |

### 4.2 单租户两种跑法

| 跑法 | compute | Volume | 合计/月 | 代价 |
|---|---|---|---|---|
| **常开**（min 1 实例不休眠） | $1.94–3.14 | $0.15 | **≈$2.1–3.4** | 无冷启动，体验最好 |
| **scale-to-zero**（闲置自动停） | $0.2–0.5（按实际在线秒） | $0.15（照收） | **≈$0.3–1** | 每次唤醒有几秒冷启动；下限被 Volume 钉住 |

> Volume 是 7×24 计费、不因停机免费——所以 scale-to-zero 的成本下限约等于 Volume 那 $0.15/月。若用 S3 做真源、Volume 只当缓存，甚至可以更激进：不挂 Volume、每次冷启动从 S3 重建（省掉 Volume 费，换更长冷启动），进一步压低闲置成本。

### 4.3 规模化成本（≈ 单租户 × 活跃租户数）

Fly 无每-app 固定费，故总成本近似线性：

| 租户数 | 全 scale-to-zero（闲置为主） | 全常开 |
|---|---|---|
| 10 | ~$3–10/月 | ~$21–34/月 |
| 100 | ~$30–100/月 | ~$210–340/月 |
| 1,000 | ~$300–1,000/月 | ~$2,100–3,400/月 |

真实形态是两者混合（活跃用户常开、沉默用户闲置），落在两栏之间，且越偏左（多数不活跃）越可能。

**省钱杠杆**：Fly 预留折扣（预付承诺换 ~40% compute 折扣，约 $36/年换 $5/月额度）——上量后值得用。

### 4.4 成本大头不在上面这张表

| 隐性成本 | 说明 |
|---|---|
| **控制面工程** | 鉴权/开通/路由/计量/计费/后台，全部从零；这是主要投入，跟服务器费不是一个量级的事 |
| **运营与合规** | 托管他人私密笔记 → 备份、误删恢复、隐私政策、数据出境、事故响应；个人自用时全不存在 |
| **对象存储** | S3/R2 存储 + 请求费（个人库通常几百 MB～数 GB，量级很小；R2 出网免费更划算） |
| **控制面自身托管** | Gateway + 控制面 DB（Postgres）的常驻成本，小规模一台小实例即可 |

**定价含义**：轻度租户单成本约 $0.3–1/月，只要订阅定价覆盖这个数并留出利润，**基础设施本身不是瓶颈**；瓶颈是工程与运营责任。

## 5. 代码调整方向

分两侧：**数据面 = Jasper 本体（改动极小）**；**控制面 = 新建服务（全新工程）**。

### 5.1 数据面：Jasper 本体（最小改动）

得益于已有的环境变量参数化 + S3 插件，改动很少：

1. **`SIGTERM` 优雅退出**（小而必要）。当前 `shutdown_signal()` 只监听 `ctrl_c`（SIGINT）（`server/src/main.rs:215-218`）。Fly auto-stop / 平台回收会发 SIGTERM，需一并监听，保证停机前把内存态/写入干净落盘，避免损坏 `config.db`/`cache.db` 或丢未落盘写入。
2. **健康检查端点**：复用 `GET /api/status`（`server/src/api.rs:68`）作为 Fly `pingEndpoint`/health check，无需新增。
3. **仅私网暴露的部署姿势**：`JASPER_HOST=0.0.0.0` 监听 Fly 私网即可（已支持）；**务必不给实例分配公网 IP**——因为内部无鉴权，公网可达等于裸奔。这是部署约束，不是代码改动，但要在文档/编排里钉死。
4. **（可选）实例侧最小防线**：即便走私网，也可加一个"共享密钥头校验"中间件（Gateway 注入 `X-Jasper-Tenant-Secret`，实例校验），做纵深防御，防私网内越权。这是一个小中间件，挂在 `router()` 里 `guard_read_only` 同层。
5. **托管层收敛插件能力面**（安全）：托管租户若可任意装 wasm 插件且 `host:http`/`host:ai` 出网，是风险与成本面。建议托管版通过配置**关闭插件安装**或**收紧能力**（禁 `host:http`、AI 走宿主统一网关计量）。wasmi 的 fuel/内存限额已在（`server/src/plugins/runtime.rs`），但出网与安装策略需按平台策略化——可能需要一个"托管模式" feature flag 或运行时开关。
6. **密钥不落明文盘**（合规）：`config.db` 明文存 `webdav_pass`/`api_key`（`server/src/config.rs`）。每租户 Volume 隔离已降低横向风险，但静态加密/密钥托管（KMS/Secrets）在托管场景应补上。若用 S3 真源，凭证可由控制面在启动时经环境变量/Fly Secrets 注入，避免长期落在租户盘上。

> 注意：以上除第 1、6 条，其余多是"部署约束 + 可选加固"，核心逻辑不动。Jasper 保持单租户、无鉴权的简单内核，靠"外面套一层 + 每租户隔离进程"获得多租户能力——这正是方案 A 的价值。

### 5.2 控制面：新建服务（主要工程量）

一个独立的小服务（可用任意栈；若想复用 Rust 生态，axum + Postgres 顺手）：

| 模块 | 职责 | 备注 |
|---|---|---|
| **鉴权/账户** | 注册、登录、会话、找回；OAuth 可选 | 全新；这是 Jasper 现在完全没有的部分 |
| **开通/回收编排** | 调 **Fly Machines API** 按租户 create/start/stop/destroy app+machine+volume | Fly 官方 `one app per customer` 有蓝图可循 |
| **路由/反代** | 认证后把请求转发到租户实例（子域名或统一网关经 flycast） | 唯一公网入口，TLS 终止在此 |
| **计量/计费** | 记录在线时长/存储/流量 → Stripe 订阅或按量 | 与 Fly 用量对账 |
| **真源与备份** | 每租户 S3 bucket/prefix 生命周期；快照/导出 | 真源在 S3，备份成本低 |
| **后台/客服** | 封禁、配额、迁移、导出用户数据 | 运营必需 |

### 5.3 数据隔离要点

- **进程级隔离**（首选，方案 A 天然给到）：每租户独立 machine + volume + S3 前缀，互不可见，爆炸半径 = 单租户。
- **凭证隔离**：Fly Secrets/KMS 注入，勿长期落租户盘。
- **网络隔离**：实例不暴露公网，仅 Gateway 经私网访问；Fly app 间默认隔离网络。

## 7. 风险与开放问题

- **安全责任升级**：从"个人自托管、风险自负"变成"你替他人保管私密数据"。鉴权、备份、误删恢复、隐私合规成为硬需求，非可选。
- **无鉴权内核的双刃**：靠"私网 + Gateway"把身份收敛到一处很省事，但一旦私网边界被突破（配置失误把实例暴露公网、SSRF 等），后果是整租户数据裸奔。必须在编排层强约束"实例永不公网"。
- **插件出网面**：`host:http`/`host:ai` 在托管多租户下是滥用/成本/合规面，需策略化（托管版默认收紧）。
- **冷启动 vs 成本**：scale-to-zero 省钱但有唤醒延迟；纯 S3 无 Volume 更省但冷启动更慢。需按实测在"体验/成本"间取一个默认档，可能按订阅档差异化（免费档 scale-to-zero，付费档常开）。
- **对账**：控制面计费与 Fly 实际用量需对账，避免亏损或超收。
- **迁移/退出**：必须提供"导出全部笔记（标准 Joplin 格式）"的能力，既是信任也是合规要求——好在数据本就是 Joplin 文件，导出成本低。

---

## 8. 结论

- **技术上可行且改动可控**：选"按量付费容器"档（首选 Fly.io Machines），走"每租户一实例 + S3 真源 + 控制面网关"，Jasper 本体几乎不动（主要加 SIGTERM 与托管加固），把多租户复杂度挡在核心之外。
- **成本量级友好**：基础设施单租户 $0.3–3.4/月，订阅定价容易覆盖。
- **真正的门槛是产品与运营**：从零建控制面（鉴权/开通/路由/计费）和承担托管他人私密数据的运营与合规责任，才是这件事的主要投入——而非服务器账单。

> 若决定推进，建议先做 P0：手工在 Fly 上起一个 Jasper 实例 + S3 真源，实测冷启动、增量缓存命中、scale-to-zero 的真实体验和账单，再据此决定控制面的技术选型与订阅定价。
